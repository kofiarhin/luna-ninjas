# Multi-Operation Math Game: Multiplication & Division

## Summary

Expand the game from multiplication-only into a two-operation math game. Before starting a round, the user chooses an operation (multiplication or division) and then a table number. The selected operation controls the entire round. Division questions use the selected table as the divisor and always produce whole-number answers. Scores from both operations contribute to the same leaderboard and use the same scoring formula. The insights/mastery system is extended to be operation-aware so that multiplication and division performance are tracked separately.

## Goal

A user can choose multiplication or division, pick a table, and complete a full round in the selected operation. Logged-in users have scores saved and insights recorded. The leaderboard remains unified. Division rounds are as polished and consistent as multiplication rounds.

## Non-Goals

- No mixed-operation rounds (a round is always one operation)
- No subtraction or addition
- No fractional division answers
- No negative numbers
- No separate leaderboard per operation
- No adaptive difficulty engine
- No AI hint system
- No guest score persistence
- No implementation code in this spec

## Problem

The game only supports multiplication, limiting its educational value. Adding division reuses the same table-based structure (tables 2–12, multipliers 1–12) and doubles the practice surface area with minimal new complexity.

## Users / Actors

- **Guest user**: can play both multiplication and division rounds; scores and insights are not saved
- **Authenticated user**: can play both; scores are saved, insights are recorded, leaderboard reflects cumulative totals

## Existing Repo Findings

### Current Game Flow

1. `/game` route renders `Game.jsx`
2. `Game.jsx` shows `TableSelector` (tables 2–12) → user picks a table → `MultiplicationGame` renders
3. `MultiplicationGame` manages the full round: 12 questions, 8s timer, 5 lives, streak bonuses
4. Questions generated by `generateRound(table)` in `utils/questionGenerator.js` — always `a = table`, `b = shuffled 1–12`, `correctAnswer = table * b`
5. `questionLog` entries: `{ id, a, b, correctAnswer, userAnswer, isCorrect, timeTaken, outcome }`
6. On game over, if user is authenticated: `submitScore({ table, correctCount })` and `record(updatedLog)` fire

### Current Score Submission

- Frontend hook `useSubmitScore` sends `POST /api/scores` with `{ table, correctCount }`
- Backend `scoreController.submitScore` validates `table` (2–12) and `correctCount` (0–12), computes `roundScore = WEIGHT_MAP[table] * correctCount`, atomically increments `User.totalScore` and `User.gamesPlayed`
- `WEIGHT_MAP`: `{ 2:1, 3:2, 4:3, ... 12:11 }` — formula is `table - 1`

### Current Insights

- Frontend hook `useRecordInsights` sends `POST /api/insights/record` with `{ facts: [{ a, b, isCorrect }] }`
- Backend `insightController.recordInsights` normalizes to `(min(a,b), max(a,b))` for commutative merging, upserts with `$inc`
- `FactMastery` schema: `{ userId, factA, factB, correct, wrong, lastSeen }`
- Compound unique index on `(userId, factA, factB)`
- Insights page renders a heatmap grid (tables 2–12 × multipliers 1–12)

### Current Leaderboard

- `GET /api/leaderboard` — public, returns top 20 by `totalScore` desc
- Name fallback: `displayName || username || fullName`; entries with no name are filtered out

## Proposed Behavior

### Operation Selection Flow

1. User navigates to `/game`
2. **Step 1 — Operation selector**: Game page shows two operation buttons: "Multiplication" and "Division"
3. **Step 2 — Table selector**: After choosing an operation, the existing `TableSelector` appears. The heading adapts to say "Choose your times table" (multiplication) or "Choose your division table" (division). The points-per-correct display remains the same since scoring is identical.
4. **Step 3 — Round**: The round plays with the selected operation. All 12 questions use that single operation.
5. "Play Again" returns to the operation selector (Step 1), not the table selector.

The operation selector is a new pre-step inside `Game.jsx`, before the table selector. It does not live inside `TableSelector` or on a separate route.

### State Flow in Game.jsx

Current: `selectedTable === null` → show TableSelector; else → show MultiplicationGame.

Proposed: `selectedOperation === null` → show OperationSelector; `selectedTable === null` → show TableSelector; else → show game component (now renamed or made operation-aware).

## Multiplication Round Rules (unchanged)

- `generateRound(table)` produces 12 questions: `a = table`, `b = shuffled 1–12`, `correctAnswer = table * b`
- Question text: `"What is {table} × {b}?"`
- Wrong answers generated relative to the correct product

## Division Round Rules (new)

- A new `generateDivisionRound(table)` function produces 12 questions
- For each multiplier `b` in shuffled `[1..12]`:
  - `dividend = table * b`
  - `divisor = table`
  - `correctAnswer = b` (the quotient)
  - Question text: `"What is {dividend} ÷ {table}?"`
- All answers are whole numbers by construction
- Wrong answers: generated relative to `correctAnswer` (the quotient), using similar offsets as multiplication wrong-answer generation (±1, ±2, etc.), filtered to positive integers only
- The `a` and `b` fields in the question object:
  - `a = dividend` (e.g., 21)
  - `b = table` (the divisor, e.g., 7)
  - This means `a` and `b` semantics differ per operation — `a` is always the first operand shown, `b` is always the second

### questionLog Entry Shape (updated)

Each entry gains an `operation` field:

```
{
  id, a, b, correctAnswer, userAnswer, isCorrect, timeTaken, outcome, operation
}
```

- `operation`: `"multiplication"` or `"division"`
- For multiplication: `a = table`, `b = multiplier`, `correctAnswer = a * b`
- For division: `a = dividend`, `b = divisor (table)`, `correctAnswer = quotient`

## Scoring Rules

**Decision: Multiplication and division use identical scoring.**

- Both use the same `WEIGHT_MAP[table]` formula: `points = (table - 1) * correctCount`
- Rationale: the table difficulty is the same regardless of operation. Dividing by 12 is as hard as multiplying by 12. Using the same formula keeps the leaderboard fair and the backend simple.
- The score submission payload adds `operation` for data integrity, but the backend scoring formula remains `WEIGHT_MAP[table] * correctCount` regardless of operation.

### Updated Score Submission Payload

```
POST /api/scores
Body: { table, correctCount, operation }
```

- `operation`: `"multiplication"` or `"division"` — validated by backend
- The backend validates `operation` is one of the two allowed values
- The scoring calculation remains `WEIGHT_MAP[table] * correctCount` — operation-agnostic
- `operation` is accepted and logged but does not change the score formula

### Why persist operation in score submission?

For future extensibility: operation-specific leaderboards, per-operation stats, or operation-weighted scoring could be added later without migrating data. For MVP, it's just validated and ignored in the calculation.

### Backend Score Controller Changes

- Accept optional `operation` field in request body
- Validate: if present, must be `"multiplication"` or `"division"`; if absent, default to `"multiplication"` for backward compatibility with any in-flight requests
- Score calculation unchanged: `WEIGHT_MAP[table] * correctCount`
- No new database fields on User model — `totalScore` and `gamesPlayed` continue to aggregate across both operations

## Insights / Mastery Impact

**Decision: Make insights operation-aware by adding an `operation` field to FactMastery.**

### Rationale

Multiplication fact `3 × 7 = 21` and division fact `21 ÷ 7 = 3` exercise different cognitive skills even though they involve the same numbers. Merging them into one record would inflate apparent mastery. Separating them gives accurate per-operation tracking.

### FactMastery Schema Change

Add `operation` field:

```
{
  userId, factA, factB, operation, correct, wrong, lastSeen
}
```

- `operation`: `"multiplication"` or `"division"`, required, default `"multiplication"`
- Compound unique index changes from `(userId, factA, factB)` to `(userId, factA, factB, operation)`

### Commutative Normalization

- **Multiplication**: continues to store as `(min(a,b), max(a,b))` — `3 × 7` and `7 × 3` are the same fact
- **Division**: store as `(min(a,b), max(a,b))` where `a = dividend`, `b = divisor`. Since `dividend > divisor` always (dividend = table * multiplier, divisor = table, multiplier ≥ 1), `factA = table` (smaller), `factB = dividend` (larger). Division is not commutative, but the `operation` field distinguishes it from the corresponding multiplication fact.

### Insight Recording Payload Change

```
POST /api/insights/record
Body: { facts: [{ a, b, isCorrect, operation }] }
```

- Backend applies the same `(min, max)` normalization plus stores the `operation` field
- If `operation` is missing, default to `"multiplication"` for backward compatibility

### Insights Page Changes

The Insights page currently shows one heatmap for multiplication. With operation-aware data:

- Add a toggle or tab at the top of the Insights page: "Multiplication" | "Division"
- Default view: Multiplication (matches existing behavior)
- Division view: same heatmap grid structure, but cells show division fact mastery
- Table summary cards: filtered by selected operation
- Heatmap corner symbol changes: `×` for multiplication, `÷` for division
- Cell tooltip adapts: `"7 × 3 — Mastered (95%)"` vs `"21 ÷ 7 — Learning (75%)"`

### Backward Compatibility for Existing Insight Data

- All existing `FactMastery` documents have no `operation` field
- The migration path: add `operation` with default `"multiplication"` in the schema
- Existing records are treated as multiplication facts
- The new compound unique index `(userId, factA, factB, operation)` must replace the old `(userId, factA, factB)` index
- This is a one-time index migration: drop the old index, create the new one. Mongoose handles this on startup if `autoIndex` is enabled (default).

## Edge Cases

1. **Old clients sending score without operation**: Backend defaults to `"multiplication"` — no breakage
2. **Old FactMastery documents without operation field**: Schema default `"multiplication"` applies on read; existing compound index must be replaced
3. **Division wrong answers must be positive integers**: Wrong answer generator for division must filter to positive integers only, since quotients are small (1–12)
4. **Division wrong answer pool is smaller than multiplication**: Quotients range 1–12 vs products up to 144. The wrong answer generator must handle this smaller range gracefully, ensuring 3 distinct wrong answers exist
5. **User switches operation mid-session via browser back**: `handlePlayAgain` resets to operation selector, so this is safe
6. **Table selector shows same tables for both operations**: Tables 2–12 are valid divisors and produce whole-number quotients with multipliers 1–12
7. **Index migration**: The compound unique index change on FactMastery requires dropping the old index. If the app uses `autoIndex: true` (Mongoose default), Mongoose will attempt to create the new index on startup. The old index must be dropped manually or via a migration script if `autoIndex` is disabled.

---

## Migration / Backward Compatibility Notes

1. **Score submission**: Adding optional `operation` field is backward-compatible. Old clients omitting it get `"multiplication"` default.
2. **FactMastery index**: The compound unique index must change from `(userId, factA, factB)` to `(userId, factA, factB, operation)`. Steps:
   - Add `operation` field with default `"multiplication"` to schema
   - Drop old index: `db.factmasteries.dropIndex("userId_1_factA_1_factB_1")`
   - New index is created by Mongoose on next startup (if autoIndex is on)
   - Existing documents get `operation: "multiplication"` via schema default
3. **Insights API**: Adding `operation` to the response is additive — existing consumers ignore unknown fields
4. **No User model migration needed**: `totalScore` and `gamesPlayed` are operation-agnostic aggregates
